# F2 Contracts — Agreements Between Features

**Feature:** F2 — Orchestration Scripts (Data Dictionary)
**Project:** Synchrony Documentation Automation Skillset
**Last Updated:** April 10, 2026

---

## What This Document Covers

Contracts define the handshake between F2 and the features it talks to. They answer: "What does F2 send, what does F2 expect back, and what happens when something goes wrong?"

This is NOT a repeat of data-model.md. Data-model.md defines what every file looks like. This document defines the **agreements** — what each feature promises to deliver and what happens if it doesn't.

---

## Contract 1: F2 → F1 (What F2 Sends to the LLM)

### What Happens

`extract_fields.py` produces `extracted_fields.json`. SKILL.md (F1) reads this file and feeds it to Claude for the LLM step.

### The Agreement

F2 promises to deliver a JSON array of objects, one per field, each with exactly 8 keys:

```json
[
  {
    "field_name": "LIMIT_BAL",
    "type": "DECIMAL",
    "nullable": false,
    "constraints": ["CHECK (LIMIT_BAL > 0)"],
    "enums": [],
    "source_path": "sample_schema.json → credit_card_clients → LIMIT_BAL",
    "schema_comments": "Credit limit in NT dollars",
    "table_name": "credit_card_clients"
  }
]
```

### F2's Guarantees

| Guarantee | Details |
|-----------|---------|
| Every field has exactly 8 keys | No missing keys, no extra keys |
| `field_name` is always a non-empty string | Validated in Step 1 |
| `source_file` is always a non-empty string | Validated in Step 1 — used to build `source_path` for each field |
| Optional fields have defaults filled in | `type` → `"UNKNOWN"`, `nullable` → `true`, `constraints` → `[]`, `enums` → `[]`, `schema_comments` → `null` |
| `source_path` is always present | Generated by `extract_fields.py` |
| File is valid JSON | Always parseable |
| File location is fixed | `output/intermediate/extracted_fields.json` |

### What F1 Should NOT Assume

- Field order is not guaranteed to match the original schema order (though it usually does)
- `type` values are not validated against real SQL types — `"UNKNOWN"` and non-standard types are possible
- `schema_comments` can be `null`

### Cross-Reference

This contract aligns with F1's input contract. See F1 contracts for F1's side of this agreement.

## Contract 2: F1 → F2 (What the LLM Sends Back)

### What Happens

Claude (via SKILL.md) reads the extracted fields and writes descriptions. The output is saved as `llm_output.json`. F2's `attach_citations.py` reads this file.

### The Agreement

F1 promises to deliver a JSON array of objects, one per field, each with exactly 5 keys:

```json
[
  {
    "field_name": "LIMIT_BAL",
    "description": "Amount of credit available to the cardholder in NT dollars",
    "confidence": "High",
    "evidence_refs": [
      "field_name: 'LIMIT_BAL' suggests credit limit or balance",
      "schema_comments: 'Credit limit in NT dollars' confirms meaning"
    ],
    "clarification_flag": false
  }
]
```

### What F2 Expects

| Field | Type | Required | Allowed Values |
|-------|------|----------|---------------|
| `field_name` | string | Yes | Should match an extracted field name |
| `description` | string | Yes | ~25 words or fewer (soft limit) |
| `confidence` | string | Yes | `"High"`, `"Medium"`, or `"Low"` |
| `evidence_refs` | array of strings | Yes | At least one entry |
| `clarification_flag` | boolean | Yes | `true` or `false` |

### What F2 Does If F1 Breaks the Agreement

F2 doesn't crash. It handles every failure case:

| Problem | How F2 Detects It | What F2 Does |
|---------|-------------------|--------------|
| File doesn't exist | Checks for `llm_output.json` | All fields get placeholders |
| File isn't valid JSON | JSON parse fails | All fields get placeholders |
| Wrong number of fields | Count check | Missing fields get placeholders, extras ignored |
| Field name doesn't match | Name comparison (case-insensitive) | Tries case-insensitive match, then position fallback |
| Field name casing is wrong | Case-insensitive match succeeds | Merges it, logs `name_case_mismatch` correction |
| `confidence` is `"Low"` but `clarification_flag` is `false` | Logic check | Overrides flag to `true`, logs `clarification_flag_override` correction |
| Description is over 25 words | Word count | Keeps it (soft limit), logs `description_over_limit` correction |
| Fields returned in wrong order | Order comparison | Matches by name, logs `order_mismatch` correction |
| Duplicate field names in output | Name matching finds two | Uses first match, ignores second, logs `duplicate_name_in_output` correction |

### Retry Behavior

- **Who retries:** F2 owns retry logic, not F1.
- **Max attempts:** 3 total (1 initial + 2 retries)
- **F2's role:** F2 validates the output file after each attempt. If the file is bad after all retries, F2 fills placeholders and moves on.

This contract aligns with F1's output contract. See F1 contracts for F1's side of this agreement.

## Contract 3: F2 → F3 (How F2 Uses Templates)

### What Happens

`assemble_output.py` reads `timestamped_fields.json` and an F3 Markdown template to produce the final `data_dictionary.md`. `generate_qa_report.py` does the same for `qa_report.md`.

### The Agreement

F3 delivers static template files. F2 reads them at runtime to determine the output format. The templates are the single source of truth for output structure (FR-F3-005) — F2 follows the templates, not the other way around.

### File Paths (Guaranteed by F3)

| F2 Script | F3 File | Guaranteed Path |
|-----------|---------|----------------|
| `assemble_output.py` | Data dictionary template | `assets/data_dictionary_template.md` |
| `generate_qa_report.py` | QA report template | `assets/qa_report_template.md` |
| All scripts (testing) | Sample schema | `assets/sample_schema.json` |
| — | Gold standard data dictionary | `assets/example_data_dictionary.md` |
| — | Gold standard QA report | `assets/example_qa_report.md` |

F2 can hardcode these paths. F3 guarantees they won't change.

### Template Format

Templates use a **structure-based** approach (F3 Decision D-001):

| Template Part | How F2 Uses It |
|---------------|---------------|
| **Header** | Simple `{placeholder}` substitution — replace `{table_name}`, `{source_file}`, `{generation_date}` with real values |
| **Main table / sections** | Read the column headers or section structure from the template, then generate content programmatically |
| **Footer** | Copy as-is into the output (static content) |

**Placeholder format:** Single curly braces — `{table_name}`, not `{{table_name}}`. Only 3 header placeholders exist. Table rows and evidence entries are NOT placeholders — F2 generates those dynamically.

### What F2 Expects from the Data Dictionary Template

| Expectation | What Breaks If Violated |
|-------------|------------------------|
| File exists at `assets/data_dictionary_template.md` | Script stops with error message |
| Valid Markdown with properly formatted tables | `assemble_output.py` can't parse the structure |
| 8 columns in the main table: field_name, type, nullable, constraints, enums, description, confidence, last_verified | Output has wrong columns or missing data |
| 3 header placeholders: `{table_name}`, `{source_file}`, `{generation_date}` | Output shows raw placeholder text |
| Evidence & Citations section with `### {field_name}` subheading pattern | Evidence section is missing or malformed |
| Footer with confidence legend | Auditors don't know what `"High"`, `"Medium"`, `"Low"` means |

### What F2 Expects from the QA Report Template

| Expectation | What Breaks If Violated |
|-------------|------------------------|
| File exists at `assets/qa_report_template.md` | Script stops with error message |
| Report Header with run metadata fields | QA report has no metadata |
| 6 sections in this exact order: Coverage Statistics, Confidence Distribution, Fields Requiring Clarification, Merge Corrections, Warnings, Processing Notes | Sections are missing, out of order, or have wrong names |

### What F2 Does If a Template Is Missing

1. Script stops immediately
2. Prints: "Template not found at `assets/{filename}`. Pipeline cannot produce formatted output without the template. Ensure all F3 asset files are present before running."
3. No output is produced for that step

**This is NOT graceful degradation.** A missing template is a setup error — someone forgot to include the file. The fix is "put the file back." This is different from the LLM going offline (which IS graceful degradation).

**Why not a hardcoded fallback template:** Creates two sources of truth. If someone updates the real template but forgets the backup, the pipeline silently uses the wrong format.

### What F2 Can Test Without F3

Steps 1 through 4 (`validate_input.py` through `add_timestamps.py`) don't use templates. You can build and test those scripts before F3 delivers anything. Steps 5 and 6 (`assemble_output.py` and `generate_qa_report.py`) need the templates.

### Cross-Reference

This contract aligns with F3's contracts document (Contract 2: F3 → F2). See F3's contracts for F3's side of this agreement.

## Contract 4: Error Contract

### The Core Rule

**The pipeline never stops with no output.** No matter what goes wrong, F2 produces both `data_dictionary.md` and `qa_report.md`. They might be full of placeholders, but they exist.

**Two exceptions where the pipeline stops entirely:**

1. **Invalid input** — `validate_input.py` rejects the schema. If the schema is broken, there's nothing useful to produce.
2. **Missing template** — `assemble_output.py` or `generate_qa_report.py` can't find its F3 template file. This is a setup error, not a runtime failure.

### Failure Scenarios

| Scenario | What F2 Does | Output Quality |
|----------|-------------|---------------|
| Input schema is invalid | `validate_input.py` rejects it with clear error messages. Pipeline stops here. | No output — but the user gets actionable error messages telling them exactly what to fix. |
| LLM returns bad output, retries succeed | F2 retries (max 2 retries, 3 total attempts). F2 processes the successful attempt normally. | Full quality. |
| LLM returns bad output, all 3 attempts fail | F2 fills all fields with placeholders. | Both deliverables produced. 0% coverage. QA report shows full failure. |
| LLM is completely offline | Same as above — `llm_output.json` doesn't exist. | Both deliverables produced. 0% coverage. |
| LLM returns partial results | F2 merges what it can, fills placeholders for the rest. | Partial coverage. QA report shows which fields were missed. |
| F3 template is missing | Script stops with error: "Template not found at `assets/{filename}`." | No output for that step. This is a setup error — fix by ensuring all F3 asset files are present. |

### Three Ways the Team Knows Something Went Wrong

1. **Console message** — Printed immediately during the run. Shows what failed, how many attempts, and what action was taken.
2. **QA Report Section 6 (Processing Notes)** — Permanent record in the output file. The console message disappears when you close the terminal; this stays.
3. **Field-level `merge_status`** — Every placeholder field has `merge_status: "placeholder"`. You can see exactly which fields were affected.

### What Does NOT Happen on Error

- No separate error log file (QA report covers this)
- No email or notification (file-based system, no cloud services)
- No crash or unhandled exception (all failure cases are handled)
- Pipeline does not silently succeed with bad data (corrections are always logged)

### Alignment Check

- **F2 → F1 and F1 → F2** contracts should match what's defined in the F1 contracts. If there's ever a conflict between this document and the F1 contracts, the team should resolve it before implementation — both sides need to agree on the exact JSON structures.
- **F2 → F3** contract should match what's defined in F3's contracts document (Contract 2: F3 → F2). If there's a conflict, the template is the source of truth (FR-F3-005) and F2's code must be updated to match.
