# Implementation Plan: F6 — RCSA Scripts (Deterministic Python Pipeline)

**Branch**: `f6-rcsa-scripts` | **Date**: 2026-04-08 | **Spec**: spec.md
**Input**: Feature specification from `/specs/f6-rcsa-scripts/spec.md`
**Version**: 2.0 — Revised after QA Audit, Edge Case Attack, and Demo Prioritization

## Summary

Build the deterministic Python pipeline that supports the RCSA Control Narrative Generation skill. The scripts handle everything outside the LLM reasoning step: input validation (JSON existence, validity, schema conformance), artifact registry building, evidence-to-control mapping using the control library, LLM invocation orchestration with retry logic and graceful degradation, citation validation against the artifact registry to detect hallucinations, and output file writing (`rcsa_control_narratives.md`, `validation_report.md`). The scripts consume the output structure defined by F5 (SKILL.md) and produce inputs that F4 (Assets) will later formalize into reusable templates and sample data.

**Deliverables (Tier 1 — Demo):**
- Input validation module (`validate_inputs.py`) — file existence, JSON validity, schema conformance, duplicate detection
- Artifact registry builder (`build_artifact_registry.py`) — structured index of all repository artifacts
- Evidence mapper (`map_evidence.py`) — maps artifacts to compliance controls using control library
- LLM orchestrator (`orchestrate_llm.py`) — invocation, retry logic, timeout handling, graceful degradation
- Citation validator (`validate_citations.py`) — checks every citation against artifact registry
- Output writer (`write_output.py`) — writes Markdown output files with directory creation
- Pipeline orchestrator (`pipeline.py`) — main entry point chaining all modules
- Environment setup with `.env` configuration and `.gitignore`
- Unit tests for all modules + integration test for full pipeline
- Sample test data (valid and invalid input sets)

**Deliverables (Tier 2 — Future/Documented):**
- Unicode normalization for artifact identifiers
- Large file handling (>10MB streaming/chunked parsing)
- Concurrent execution file locking
- Windows-specific path length handling
- Output file versioning (timestamped directories)
- Context window size management for LLM input
- Cross-file semantic validation
- PII filtering before LLM invocation
- Formal exit code strategy for CI/CD integration

All Tier 1 modules must pass unit tests and the full integration test. Target: May 7, 2026 demo.

### Pre-Flight Check

Before any pipeline run (development or demo), execute a minimal API validation:

1. Load API key from .env using `python-dotenv`
2. Verify `ANTHROPIC_API_KEY` is set and non-empty
3. Make a lightweight Claude API call (e.g., "respond with OK")
4. If success → proceed with pipeline
5. If failure → exit immediately with clear error: "API key invalid or expired — check your .env configuration"

This catches expired keys, billing limits, or misconfigured environments before the full pipeline runs. Auth failures (401/403) are **never retried** — they fail immediately.

## LLM Resilience Plan

**Owner**: F6 team — implemented in `orchestrate_llm.py`.

### Retry Logic

| Scenario | Behavior | Max Retries | Backoff |
|----------|----------|-------------|---------|
| Rate limit (HTTP 429) | Retry with exponential backoff + jitter (±20%) | 3 | 1s → 2s → 4s |
| Timeout (>30s) | Retry once | 1 | Immediate |
| Empty/null response | Retry with simplified prompt | 2 | 1s → 2s |
| Server error (500/502/503) | Retry with exponential backoff | 3 | 1s → 2s → 4s |
| Auth failure (401/403) | **Do not retry** — fail immediately | 0 | N/A |
| Malformed response | **Do not retry** — log raw response, fall back to degradation | 0 | N/A |

### Error Classification

The orchestrator must classify errors into distinct categories and handle each differently:

- **Auth errors (401/403)**: Never retry. Fail immediately with clear message about `.env` configuration.
- **Rate limits (429)**: Retry with exponential backoff. These are transient and expected.
- **Server errors (5xx)**: Retry with exponential backoff. These are transient infrastructure issues.
- **Timeouts**: Retry once. If the LLM is consistently slow, degradation is better than waiting.
- **Empty responses**: Retry with simplified prompt. May indicate context window issues.
- **Malformed responses**: Do not retry (same prompt will likely produce same result). Log and degrade.

### Graceful Degradation (Constitution Principle 2)

On LLM failure after all retries exhausted:

1. Scripts output raw extracted data (evidence mapping, artifact list) without narrative
2. Output file uses distinct filename: `rcsa_control_narratives.DEGRADED.md`
3. Output file includes header: `⚠️ Generated in degraded mode — LLM unavailable. Raw data preserved for manual narrative writing.`
4. Citation validation is skipped (no LLM output to validate)
5. Validation report notes: "LLM unavailable — citation validation not performed"
6. Team can manually write short narratives using extracted data as input

### Error Messages

| Scenario | User-Facing Message |
|----------|---------------------|
| Rate limit | "API rate limit reached. Retrying in {n} seconds... (attempt {x}/3)" |
| Timeout | "LLM response timed out after 30s. Retrying..." |
| Auth failure | "API key invalid or expired. Check your .env configuration." |
| Server error | "Claude API server error ({code}). Retrying in {n} seconds... (attempt {x}/3)" |
| Empty response | "LLM returned empty response. Retrying with simplified prompt... (attempt {x}/2)" |
| Malformed response | "LLM response format unexpected. Falling back to degraded output. Raw response logged for debugging." |
| All retries exhausted | "LLM unavailable after {n} attempts. Generating degraded output with raw data only." |
| Missing API key | "ANTHROPIC_API_KEY not found in environment. Create a .env file from .env.example and add your API key." |

## Tier 2: Future / Documented (Not Built for Demo)

These items are documented for the roadmap and Synchrony handoff. They are **not** required for the May 7th demo.

| Item | Rationale | Handoff Note |
|------|-----------|--------------|
| Unicode normalization for artifact identifiers | Demo uses clean, hand-crafted identifiers. Real-world repos may have Unicode edge cases. | "In production, we'd normalize all identifiers using `unicodedata.normalize('NFKC')` and strip zero-width characters to prevent silent citation mismatches." |
| Large file handling (>10MB artifact index) | Demo uses small sample data. Enterprise-scale is out of scope. | "In production, we'd add streaming JSON parsing and memory guards for large repositories." |
| Concurrent pipeline execution / file locking | Demo is single-user. No risk of concurrent runs. | "In production, we'd add file locking to prevent output corruption from parallel runs." |
| Windows-specific path length limits (>260 chars) | Demo runs on macOS/Linux. Windows support is a future concern. | "We've documented Windows path length as a known limitation for future cross-platform support." |
| Symlink handling / path traversal protection | Demo uses known, controlled input files. | "In a shared environment, we'd add path validation to prevent directory traversal." |
| Output file versioning (timestamped directories) | Demo runs once or twice. No risk of losing previous output. | "In production, we'd add timestamped output directories to preserve run history." |
| Context window size management | Demo-scale data fits in context. | "For enterprise-scale repositories, we'd chunk mapped evidence by control to stay within context limits." |
| Cross-file semantic validation | Schema validation catches structural issues. Semantic validation is deeper. | "In production, we'd add cross-file consistency checks between the artifact index and control library." |
| PII filtering before LLM invocation | Demo uses sanitized sample data. No PII risk. | "In production, we'd add a pre-send filter to detect and redact potential PII before LLM invocation — per our Constitution's Never-Ever Rules." |
| Partial output rollback (transactional writes) | Low probability during demo. | "In production, we'd write to a temp directory first and move to final location only on success." |
| Formal exit codes for CI/CD | Demo shows console output. Formal exit codes are for CI/CD integration. | "For CI/CD integration, we'd add structured exit codes for each failure mode." |
| LLM model version pinning | Low risk over 4 weeks. | "In production, we'd pin the Claude model version to prevent unexpected output changes." |
| `max_tokens` truncation detection | Mitigated by setting generous `max_tokens` value. | "In production, we'd detect truncated responses and re-request with adjusted parameters." |
