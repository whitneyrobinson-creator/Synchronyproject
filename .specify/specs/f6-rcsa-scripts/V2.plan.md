## Summary

Build the assets package that formalizes the output templates, sample input data, control library, and citation format specification for the RCSA Control Narrative Generation skill. F4 is the final feature built — it takes the output structures defined inline by F5 (SKILL.md) and the pipeline requirements established by F6 (Scripts) and packages them into reusable, versioned asset files.

**Deliverables (Tier 1 — Demo):**
- Two Markdown output templates (`rcsa_control_narratives_template.md`, `validation_report_template.md`)
- Three JSON sample input files (`sample_artifact_index.json`, `sample_test_catalog.json`, `control_library.json`)
- Environment setup with `.env` configuration
- LLM resilience handling (retry logic, timeout, graceful degradation)

**Deliverables (Tier 2 — Future/Documented):**
- Citation format specification (`citation_format_spec.md`)
- JSON Schema validation files
- Asset versioning strategy
- New control maintenance documentation

All Tier 1 assets must be backward-compatible with F5 and F6 and pass F6's input validation with 0 errors. Target: May 7, 2026 demo.

## Technical Context

**Language/Version**: Markdown (output templates) and JSON (sample data, control library) — no executable code in F4 assets. LLM resilience logic is Python 3.11 (coordinated with F6).
**Primary Dependencies**: F5 (SKILL.md) defines the output structure that templates formalize; F6 (Scripts) defines the input schemas that sample data must conform to, the citation format the spec must match, and the LLM interaction layer where retry/timeout logic lives. Both are upstream dependencies — F4 formalizes what they established.
**Storage**: Local filesystem — all assets stored as static files in `.cursor/skills/rcsa/assets/` (Constitution constraint: no database, no cloud storage)
**Testing**: Manual validation + automated checks — verify templates match F5/F6 output structure via diff; run sample data through F6's input validation script to confirm 0 errors; verify control library schema matches F6's expected format; pre-flight API key check
**Target Platform**: Claude Agent Skills (Cursor IDE integration) — assets are consumed by the SKILL.md (F5) and scripts (F6) at runtime

**Project Type**: Static asset package (documentation and data files — no executable code) + LLM resilience coordination with F6
**Performance Goals**: N/A — assets are static files with no runtime performance requirements. LLM calls must timeout at 30 seconds max.
**Constraints**: Templates must not introduce new formatting or sections that F5 and F6 don't already support (backward compatibility); all JSON files must use UTF-8 encoding (no BOM); sample data must exercise both narrative generation and gap detection scenarios; API keys must never be hardcoded
**Scale/Scope**: Demo-scale — sample data represents a small repository sufficient for the May 7th demo, not enterprise-scale

- [ ] Create `.env` file from `.env.example` for local configuration

- [ ] Add `.env` to `.gitignore` — verify before every commit
- [ ] Add `.env.example` to the repo with placeholder values (no real keys)

### Pre-Flight Check

Before any pipeline run (development or demo), execute a minimal API validation:

```
1. Load API key from .env
2. Make a lightweight Claude API call (e.g., "respond with OK")
3. If success → proceed with pipeline
4. If failure → exit immediately with clear error: "API key invalid — check your .env configuration"
```

This catches expired keys, billing limits, or misconfigured environments before the full pipeline runs.

## Sample Data Design Matrix

**Decision**: Define upfront which controls have evidence and which don't, to ensure the demo exercises both narrative generation and gap detection (SC-003).

| Control | ID | Evidence in Sample Data | Demo Purpose |
|---------|-----|------------------------|--------------|
| Access Control | AC | ✅ Sufficient (≥2 matching artifacts) | Shows full narrative generation with citations |
| Change Management | CM | ✅ Sufficient (≥2 matching artifacts) | Shows full narrative generation with citations |
| Data Quality | DQ | ⚠️ Partial (1 matching artifact) | Shows partial narrative with gap flag for missing evidence types |
| Incident Handling | IH | ❌ None (0 matching artifacts) | Shows full gap detection — no narrative, clear gap flag |

This guarantees:
- ≥2 controls with sufficient evidence (AC, CM) → narrative generation demo
- ≥1 control with intentionally missing evidence (IH) → gap detection demo
- 1 control with partial evidence (DQ) → shows nuanced gap flagging

## LLM Resilience Plan

**Owner**: Coordinated with F6 (Scripts) team — F4 documents requirements, F6 implements.

### Retry Logic

| Scenario | Behavior | Max Retries | Backoff |
|----------|----------|-------------|---------|
| Rate limit (HTTP 429) | Retry with exponential backoff | 3 | 1s → 2s → 4s |
| Timeout (>30s) | Retry once | 1 | Immediate |
| Empty/null response | Retry with simplified prompt | 2 | 1s → 2s |
| Auth failure (401/403) | **Do not retry** — fail immediately | 0 | N/A |

### Graceful Degradation (Constitution Principle 2)

On LLM failure after all retries exhausted:
1. Scripts output raw extracted data (evidence mapping, artifact list) without narrative
2. Output file includes header: `⚠️ Generated in degraded mode — LLM unavailable. Raw data preserved for manual narrative writing.`
3. F6 validation still runs on degraded output
4. Team can manually write short narratives using extracted data as input

### Error Messages

| Scenario | User-Facing Message |
|----------|-------------------|
| Rate limit | "API rate limit reached. Retrying in {n} seconds... (attempt {x}/3)" |
| Timeout | "LLM response timed out after 30s. Retrying..." |
| Auth failure | "API key invalid or expired. Check your .env configuration." |
| Empty response | "LLM returned empty response. Retrying with simplified prompt..." |
| All retries exhausted | "LLM unavailable after {n} attempts. Generating degraded output with raw data only." |

## Edge Cases — Addressed

| Edge Case | How Addressed | Phase |
|-----------|--------------|-------|
| Control library has <4 controls | F6 validator checks `len(controls) == 4`; fails with message naming missing control(s) | Phase 1 (Task 1.4) |
| Sample JSON is valid but wrong schema | F6 validator checks required keys and value types, not just JSON parse-ability | Phase 2 (Task 2.3) |
| Duplicate control IDs in JSON | F6 validator checks uniqueness; parse as list to detect duplicates before silent merge | Phase 2 (Task 2.3) |
| All tests passing (no gap scenario) | Sample data design matrix guarantees IH has 0 evidence, DQ has partial | Phase 2 (Task 2.4) |
| Template doesn't match F5/F6 output | Templates built FROM actual output, then diffed — zero structural differences required | Phase 3 (Tasks 3.1–3.4) |
| Citation format spec ≠ F6 validator | Citation spec includes F6's validation regex verbatim — single source of truth | Phase 4 (Task 4.4) |
| LLM rate limit / timeout / empty response | Retry logic with exponential backoff + graceful degradation | Phase 4 (Task 4.3) |
| API key expired before demo | Pre-flight check at pipeline start; test key weekly; backup key available | Phase 4 (Task 4.2) |
| UTF-8 BOM in files | Explicit check for BOM-free UTF-8 in Phase 3 | Phase 3 (Task 3.6) |
| Template modification causes regression | Any template change triggers full pipeline re-test (documented as maintenance requirement) | Phase 3 Gate |
